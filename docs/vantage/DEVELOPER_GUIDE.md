# VANTAGE-Bench Developer Guide

Covers the complete picture for participants and contributors: file-to-file connections, every CLI flag, every environment variable, all three model-registration paths, and the exact commands for every benchmark task.

---

## 1. File-to-File Map

```
run.py
├── vlmeval/config.py              ← supported_VLM dict  (model name → callable)
│   ├── vlmeval/vlm/*.py           ← local model classes  (Cosmos, CosmosHF, …)
│   └── vlmeval/api/*.py           ← API model classes    (GPT4V, CosmosReason2, …)
│
├── vlmeval/dataset/video_dataset_config.py   ← supported_video_datasets dict
│   └── vlmeval/dataset/vantage_*.py          ← VANTAGE video dataset classes
│       └── vlmeval/dataset/utils/vantagebench/
│           ├── emit.py            ← writes *_submission.jsonl during evaluate()
│           ├── id_rules.py        ← canonical submission-ID format per task
│           └── adapter_*.py       ← per-task answer/format adapters
│
├── vlmeval/dataset/__init__.py    ← IMAGE_DATASET list + build_dataset()
│   └── vlmeval/dataset/vantage2d/
│       ├── datasets.yaml          ← path config loaded by load_dataset_config()
│       ├── grounding_2d_dataset.py
│       └── astro_2d_dataset.py
│
├── vlmeval/inference.py           ← infer_data_job()  (image datasets)
├── vlmeval/inference_video.py     ← infer_data_job_video()  (video datasets)
└── vlmeval/smp/                   ← utilities: file I/O, load_env(), logging
    ├── file.py                    ← LMUDataRoot() resolves dataset paths
    └── misc.py                    ← load_env() reads .env → os.environ
```

### Data path resolution

```
LMUDataRoot()  (vlmeval/smp/file.py:69)
  1. $LMUData env var (if set and exists)
  2. ~/LMUData  (default)
  override: --lmudata-root CLI flag sets $LMUData before any dataset is built
```

### Dataset lookup path

```
run.py:build_dataset(name)
  → vlmeval/dataset/__init__.py:build_dataset()
      Video tasks:  supported_video_datasets[name]  (video_dataset_config.py)
      Image tasks:  match name against IMAGE_DATASET classes
          2D image: load_dataset_config()  →  vantage2d/datasets.yaml
```

### Submission-file flow

```
run.py
  → dataset.evaluate(result_file)          (e.g. vantage_vqa.py)
      → vantagebench/emit.py:emit_submission()
          → id_rules.py:make_id()          ← formats the canonical record ID
          writes:  <work-dir>/<model>/<eval-id>/<model>_<dataset>_submission.jsonl

scripts/package_submission.py
  scans <work-dir> for *_submission.jsonl
  → writes submission.tar.gz
      → upload to https://vantage-bench.org/submit
```

### Organizer leaderboard flow

```
outputs/<model>/<eval-id>/          ← generated by run.py
  → parser/get_all_outputs.py       ← compiles per-model JSON metrics into vlmevalkit_outputs.json
  → parser/prepare_outputs_hf.py    ← builds hf/leaderboard.json (leaderboard schema)
  → hf/up.py                        ← uploads to HF Space
  → hf/app.py (Gradio)              ← renders the live leaderboard
```

## 2. CLI Flags (`run.py`)

### Input selection (required — one of the two forms)

| Flag | Type | Description |
|------|------|-------------|
| `--data` | `str …` | One or more dataset keys, e.g. `VANTAGE_VQA_8frame VANTAGE_Temporal_8frame` |
| `--model` | `str …` | One or more model keys from `supported_VLM` in `vlmeval/config.py` |
| `--config` | `str` | Path to a JSON config file. Mutually exclusive with `--data`/`--model`. |
| `config_pos` | `str` (positional) | Shorthand: `python run.py my_config.json` is equivalent to `--config my_config.json` |

### Paths

| Flag | Default | Description |
|------|---------|-------------|
| `--work-dir` | `./outputs` | Root directory for all output files. Overridden by `$MMEVAL_ROOT`. |
| `--lmudata-root` | `None` | Override the dataset root. Sets `$LMUData` before any dataset is built. |

### Run mode

| Flag | Default | Choices | Description |
|------|---------|---------|-------------|
| `--mode` | `all` | `all`, `infer`, `eval` | `all` runs inference then evaluation. `infer` stops after writing predictions. `eval` re-evaluates an existing prediction file. |

### API / inference settings

| Flag | Default | Description |
|------|---------|-------------|
| `--api-nproc` | `4` | Number of parallel workers for API model calls and LLM judge calls. |
| `--retry` | `None` | Override the retry count for API VLMs and the judge LLM. Default is 3 if unset. |
| `--use-vllm` | `False` | Pass `use_vllm=True` to the model constructor when using `--config`. Also sets `judge_kwargs['use_vllm']`. |

### Evaluation / judging

| Flag | Default | Description |
|------|---------|-------------|
| `--judge` | `None` | Force a specific judge model name (e.g. `gpt-4o`). When unset, `run.py` picks the judge automatically by dataset type. |
| `--judge-args` | `None` | JSON string of extra kwargs forwarded to the judge, e.g. `'{"temperature": 0}'`. |
| `--use-verifier` | `False` | Sets `use_verifier=True` in judge kwargs (used by some upstream datasets). |

### Resume / caching

| Flag | Default | Description |
|------|---------|-------------|
| `--reuse` | `False` | Reuse the latest existing prediction and temp pickle files from a prior run. |
| `--reuse-aux` | `True` | When `--reuse` is set, also reuse auxiliary evaluation files (parse caches, etc.). |
| `--ignore` | `False` | Ignore failed indices from a previous interrupted run instead of re-attempting them. |

### Logging

| Flag | Default | Description |
|------|---------|-------------|
| `--verbose` | `False` | Enable verbose logging for API calls and inference steps. |

## 3. Environment Variables

### Dataset paths

| Variable | Default | Description |
|----------|---------|-------------|
| `LMUData` | `~/LMUData` | Root of the LMUData layout. Datasets live under `$LMUData/datasets/<name>/`. Overridden by `--lmudata-root`. |
| `MMEVAL_ROOT` | — | If set, overrides `--work-dir` for the output directory. |

### API keys

| Variable | Description |
|----------|-------------|
| `OPENAI_API_KEY` | Required for GPT-4o, chatgpt-0125 judge, and any OpenAI-compatible model. |
| `OPENAI_API_BASE` | Base URL for an OpenAI-compatible endpoint (proxy, Azure, NIM, etc.). |
| `AZURE_OPENAI_API_KEY` | Azure-specific key (falls back from `OPENAI_API_KEY`). |
| `O1_API_KEY` | Key for `o1`, `o3`, `o4-mini` registered entries in `config.py`. |
| `O1_API_BASE` | Base URL for o1-family models. |
| `HF_TOKEN` | HuggingFace token for gated model downloads. |
| `HF_HUB_CACHE` | Override the HuggingFace Hub cache directory. |

### Inference routing

| Variable | Description |
|----------|-------------|
| `FWD_API=1` | Force all registered API models to use the `GPT4V` class (forwards through a compatible endpoint set in `OPENAI_API_BASE`). |
| `EVAL_PROXY` | HTTP proxy used only during the evaluation (judge) step; not applied during inference. |
| `LOCAL_LLM` | Used by some upstream dataset evaluators to specify a local LLM judge path. Not used by VANTAGE evaluators. |
| `CUDA_VISIBLE_DEVICES` | Restrict which GPUs are used. `run.py` reads this at startup to compute per-process GPU assignment under `torchrun`. |
| `DIST_TIMEOUT` | Seconds before distributed `dist.barrier()` calls time out. Default 3600. |

### `.env` file

`run.py` calls `load_env()` before `main()`, which reads a `.env` file at the repo root and injects any defined keys into `os.environ`. This is the easiest way to store API keys without setting them in the shell:

```bash
# .env  (never commit this file)
OPENAI_API_KEY=sk-...
OPENAI_API_BASE=https://your-proxy.example.com/v1
```

## 4. Model Registration

There are three ways to tell `run.py` which model to use.

---

### Way 1 — Pre-registered key in `config.py` (simplest)

`vlmeval/config.py` contains a `supported_VLM` dict. Every key in that dict is a valid `--model` name. No extra config needed.

```bash
# Use a pre-registered GPT-4o entry
python run.py \
  --model GPT4o \
  --data VANTAGE_VQA_8frame \
  --work-dir ./outputs

# Use a pre-registered Cosmos entry
python run.py \
  --model Cosmos-Reason1-7B \
  --data VANTAGE_VQA_8frame VANTAGE_Temporal_8frame \
  --work-dir ./outputs
```

The pre-registered NVIDIA / Cosmos entries:

| Key | Class | Notes |
|-----|-------|-------|
| `Cosmos-Reason1-7B` | `Cosmos` (vLLM) | HF: `nvidia/Cosmos-Reason1-7B` |
| `Cosmos-Reason2-8B` | `Cosmos` (vLLM) | HF: `nvidia/Cosmos-Reason2-8B` |
| `Cosmos-Reason2-8B-HF` | `CosmosHF` (transformers) | No vLLM required |
| `Cosmos-Reason2-32B` | `Cosmos` (vLLM) | Requires multi-GPU |
| `Cosmos-Reason2-8B-API` | `CosmosReason2` | NIM API endpoint |
| `Cosmos3-Nano` | `Cosmos3` (transformers) | HF: `nvidia/Cosmos3-Nano` |
| `Cosmos3-Nano-vLLM` | `Cosmos3vLLM` | Needs pre-converted checkpoint |

---

### Way 2 — JSON config file (flexible, recommended for custom params)

Pass a `--config path/to/config.json`. The config supports arbitrary model parameters and multiple model × dataset combinations in one invocation.

```json
{
  "model": {
    "MyCosmos": {
      "class": "Cosmos",
      "model_path": "nvidia/Cosmos-Reason2-8B",
      "tensor_parallel_size": 2,
      "max_model_len": 32768,
      "gpu_memory_utilization": 0.90,
      "enable_thinking": true,
      "temperature": 0.6,
      "max_new_tokens": 4096
    }
  },
  "data": {
    "VANTAGE_VQA_8frame": {},
    "VANTAGE_Temporal_8frame": {}
  }
}
```

```bash
python run.py --config my_config.json --work-dir ./outputs
```

For an empty `{}` value in `data`, the key must exist in `supported_video_datasets` (i.e., the dataset variant pre-registered in `video_dataset_config.py`). For a non-empty dict, provide at least `class` plus the dataset class's constructor kwargs.

All classes in `vlmeval.vlm` and `vlmeval.api` are available via `"class"`. Run `python -c "import vlmeval.vlm; print(dir(vlmeval.vlm))"` to see the full list.

**Config-only parameters** (available only via `--config`, ignored under `--data`/`--model`):

| Model kwarg | Description |
|-------------|-------------|
| `tensor_parallel_size` | Number of GPUs for vLLM tensor parallelism |
| `max_model_len` | Maximum sequence length for vLLM |
| `gpu_memory_utilization` | Fraction of GPU memory vLLM may use (0.0–1.0) |
| `enable_thinking` | Toggle chain-of-thought for Cosmos Reason models |
| `temperature` | Sampling temperature |
| `max_new_tokens` | Maximum output tokens |
| `top_p` | Nucleus sampling p |
| `repetition_penalty` | Repetition penalty |
| `max_pixels` | Pixel cap per image/frame passed to the model |

---

### Way 3 — vLLM server mode (`--use-vllm`)

For models in `supported_VLM` whose constructor accepts `use_vllm`, pass `--use-vllm` on the command line. This is equivalent to adding `"use_vllm": true` in the JSON config.

```bash
# Launch inference with vLLM enabled for a pre-registered key
python run.py \
  --model Cosmos-Reason2-8B \
  --data VANTAGE_VQA_8frame \
  --use-vllm \
  --work-dir ./outputs
```

Note: `Cosmos-Reason1-7B` and `Cosmos-Reason2-*` are already registered with `use_vllm=True` in `config.py`, so `--use-vllm` is redundant for those. It is mainly useful when using `--config` with a model that supports both transformers and vLLM backends.

---

### Multi-GPU with `torchrun` (data-parallel inference)

For large local models on multi-GPU nodes:

```bash
# 4 GPUs, 4 parallel inference workers
torchrun --nproc-per-node=4 run.py \
  --model Cosmos-Reason2-8B \
  --data VANTAGE_VQA_8frame \
  --work-dir ./outputs
```

`run.py` reads `LOCAL_WORLD_SIZE` and `LOCAL_RANK` from the `torchrun` environment and assigns `CUDA_VISIBLE_DEVICES` per process automatically.

**This is data parallelism** (each worker handles a shard of the dataset). For tensor parallelism (splitting a single large model across GPUs) use `tensor_parallel_size` in a JSON config instead.

---

### API models (no GPU required)

```bash
export OPENAI_API_KEY=sk-...

python run.py \
  --model GPT4o \
  --data VANTAGE_VQA_8frame VANTAGE_EventVerification_8frame \
  --api-nproc 8 \
  --work-dir ./outputs
```

`--api-nproc` controls how many concurrent requests are made. Increase it for faster throughput on large datasets. Rate-limit errors will trigger automatic retries (controlled by `--retry`).

## 5. Exact Run Commands

Replace `<model>` with your model key (pre-registered or config-file alias).

### Infer only (no evaluation)

```bash
python run.py --model <model> --data <dataset_key> --mode infer --work-dir ./outputs
```

### Evaluate only (re-score existing predictions)

```bash
python run.py --model <model> --data <dataset_key> --mode eval --work-dir ./outputs --reuse
```

### Full run (infer + eval, resume-safe)

```bash
python run.py --model <model> --data <dataset_key> --mode all --reuse --work-dir ./outputs
```

## 6. Output Structure

```
outputs/
└── <model>/
    ├── <model>_VANTAGE_VQA_8frame.xlsx          ← symlink → latest eval-id
    ├── <model>_VANTAGE_VQA_8frame_submission.jsonl   ← symlink
    └── T<date>_G<commit>/                        ← timestamped eval-id folder
        ├── <model>_VANTAGE_VQA_8frame.tsv        ← raw predictions
        ├── <model>_VANTAGE_VQA_8frame.xlsx       ← eval results
        ├── <model>_VANTAGE_VQA_8frame_submission.jsonl
        ├── model_config.txt                      ← model __dict__ dump
        └── VANTAGE_VQA_8frame_config.json        ← dataset config dump
```

The eval-id format is `T<YYYYMMDD>_G<8-char-git-hash>`. When `--reuse` is set, `run.py` copies or symlinks prediction files from the most recent matching eval-id rather than re-running inference.
